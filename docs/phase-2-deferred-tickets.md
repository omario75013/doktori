## DOKTORI-1P-DEBT — Doktori 1Password integration is non-functional

**Discovered:** 2026-05-05 during Phase 2 #21 Tasks 10-11 (REDIS_PASSWORD provisioning).

**Symptom:** `/opt/doktori/.env.tpl` references `op://Dartank-Infra/Doktori Prod/FIELD` paths but no item titled `Doktori Prod` exists in the vault. The convention used elsewhere in the same vault is **one item per secret** (e.g. `Doktori Prod - DEEPL_API_KEY`, `B2B Prod - STRIPE_SECRET_KEY`). The current `/opt/doktori/.env` is therefore NOT generated by `op inject` — it's a static file with hardcoded values written manually.

**Items currently missing from vault:**
- `Doktori Prod - DATABASE_URL`
- `Doktori Prod - NEXTAUTH_SECRET`
- `Doktori Prod - MEILISEARCH_KEY`
- `Doktori Prod - CRON_SECRET`
- `Doktori Prod - SOCKETIO_BROADCAST_SECRET`
- `Doktori Prod - POSTGRES_PASSWORD`
- (Also missing: `R2_*`, `TWILIO_*` — full audit needed)

Only `Doktori Prod - DEEPL_API_KEY` exists.

**Implication for #21 Redis:** Couldn't follow the spec's "add `op://` reference to `.env.tpl`, run `op inject`" flow because that flow has never worked for Doktori. Wrote `REDIS_PASSWORD` directly to `/opt/doktori/.env` (chmod 600) following the convention actually used by the other Doktori secrets. The 1Password backup item was attempted but the prod service account has read-only permission on the vault — Omar must create `Doktori Prod - REDIS_PASSWORD` from his desktop 1Password app to back up the password.

**Action needed:**
1. Audit current `/opt/doktori/.env` and create one 1Password item per secret it contains, using the `Doktori Prod - <SECRET_NAME>` naming convention.
2. Rewrite `/opt/doktori/.env.tpl` to use the correct paths: `KEY=op://Dartank-Infra/Doktori Prod - KEY/password`.
3. Add `op inject` to the deploy procedure (rsync + op inject + docker compose build).
4. Verify the `.env` is correctly regenerated; chmod 600.
5. Document the convention in `docs/operations/secrets.md`.

**Estimated effort:** 1-2h (mostly mechanical: enumerate secrets, create items, test).

**Priority:** Medium. The current setup works, but secret rotation requires manual SSH and is not auditable. Should be done before adding more secrets (e.g. KONNECT_API_KEY for #2 Paiement).
